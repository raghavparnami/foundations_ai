# v0.4 — Enterprise Wiki (multi-source knowledge base)

**Status**: in-progress · Phase A in flight
**Target outcome**: a `/wiki` surface that interlinks Loom's catalog (Tables),
user-uploaded documents, GitLab repos, and SharePoint sites into one
LLM-curated, human-editable, version-controlled wiki. Same retrieval target
the agent uses on chat, also browsable directly by humans (execs, analysts).

## Why a wiki

- Karpathy's "LLM-built wiki" pattern, applied to structured + unstructured
  corporate knowledge: distilled, interlinked, regenerated as sources change.
- Today Loom's per-table docs are leaves with no cross-references. Concepts
  ("Production lifecycle", "Quality methodology") are implicit in the agent's
  head — making them explicit pages makes them browsable AND raises the
  retrieval quality for both humans and the agent.
- Treating Tables / Documents / GitLab / SharePoint as homogeneous wiki
  sources lets us answer questions that span them ("show me the runbook for
  X *and* the latest quality data for that line").

## Out of scope for v0.4

- A WYSIWYG editor — markdown editing only (textarea + render).
- Auth / SSO — single-tenant for now; per-page ACL stubs but no enforcement.
- Real-time collaboration / OT — single-writer.
- Public sharing of wiki pages.

## Schema

All new tables in the `loom_catalog` DB:

```
wiki_sources (
  id            SERIAL PK,
  kind          TEXT NOT NULL,        -- 'tables' | 'documents' | 'gitlab' | 'sharepoint'
  name          TEXT NOT NULL UNIQUE,
  config        JSONB NOT NULL,       -- connector-specific (URL, token-ref, repo, etc.)
  status        TEXT NOT NULL DEFAULT 'idle',  -- 'idle'|'syncing'|'error'
  last_synced_at TIMESTAMPTZ,
  last_error    TEXT,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

wiki_documents (                       -- raw ingested artefacts before pageification
  id            SERIAL PK,
  source_id     INTEGER NOT NULL REFERENCES wiki_sources(id) ON DELETE CASCADE,
  external_id   TEXT NOT NULL,         -- table_id / file path / commit:path / Graph itemId
  title         TEXT NOT NULL,
  body_md       TEXT NOT NULL,
  content_hash  TEXT NOT NULL,
  metadata      JSONB NOT NULL DEFAULT '{}',
  fetched_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (source_id, external_id)
);

wiki_chunks (                          -- for retrieval (hybrid FTS + vector)
  id            SERIAL PK,
  document_id   INTEGER NOT NULL REFERENCES wiki_documents(id) ON DELETE CASCADE,
  ord           INTEGER NOT NULL,
  body          TEXT NOT NULL,
  tsv           tsvector,
  embedding     vector(1536),          -- nullable; populated only if OPENAI_API_KEY set
  UNIQUE (document_id, ord)
);
CREATE INDEX wiki_chunks_tsv_idx ON wiki_chunks USING GIN(tsv);
CREATE INDEX wiki_chunks_emb_idx ON wiki_chunks USING ivfflat (embedding vector_cosine_ops);

wiki_pages (                           -- the wiki itself; generated or human-authored
  id            SERIAL PK,
  slug          TEXT NOT NULL UNIQUE,
  kind          TEXT NOT NULL,         -- 'index' | 'concept' | 'leaf' | 'human'
  source_id     INTEGER REFERENCES wiki_sources(id) ON DELETE SET NULL,
  title         TEXT NOT NULL,
  body_md       TEXT NOT NULL,
  content_hash  TEXT NOT NULL,
  provenance    JSONB NOT NULL DEFAULT '{}',  -- {sources: [doc_id, ...], by: 'claude'|'human', model}
  status        TEXT NOT NULL DEFAULT 'ready',-- 'draft'|'ready'|'stale'
  tsv           tsvector,
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX wiki_pages_tsv_idx ON wiki_pages USING GIN(tsv);

wiki_links (                           -- directed edges; populated from [[slug]] in body_md
  id            SERIAL PK,
  from_page     INTEGER NOT NULL REFERENCES wiki_pages(id) ON DELETE CASCADE,
  to_slug       TEXT NOT NULL,         -- soft ref (target may not exist yet)
  to_page       INTEGER REFERENCES wiki_pages(id) ON DELETE SET NULL,
  context       TEXT,                  -- the prose around the link
  UNIQUE (from_page, to_slug)
);
CREATE INDEX wiki_links_to_idx ON wiki_links(to_page);

wiki_jobs (                            -- async ingest/regen tracking
  id            SERIAL PK,
  source_id     INTEGER REFERENCES wiki_sources(id) ON DELETE CASCADE,
  kind          TEXT NOT NULL,         -- 'ingest' | 'regen_concepts' | 'regen_index'
  status        TEXT NOT NULL DEFAULT 'queued', -- 'queued'|'running'|'done'|'failed'
  payload       JSONB NOT NULL DEFAULT '{}',
  result        JSONB,
  error         TEXT,
  started_at    TIMESTAMPTZ,
  finished_at   TIMESTAMPTZ,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

All `tsv` columns are maintained via triggers from `to_tsvector('english', body)`.

## Architecture

```
┌────────────────────────────────────────────────────────────────────┐
│                       Source connectors                            │
│  TablesIngestor   DocumentsIngestor  GitLabIngestor  SharePointMCP │
└──────────┬───────────┬──────────────┬─────────────┬─────────────────┘
           │           │              │             │
           v           v              v             v
                  wiki_documents  ← raw normalized artefacts
                        │
                        v
                  wiki_chunks  ← chunked + FTS + (optional) embedded
                        │
                        v
              [ Loop 5: ConceptBuilder ]
                        │
                        v
                  wiki_pages  ← index + concept + leaf pages
                  wiki_links  ← cross-refs from [[slug]] resolution
                        │
                        v
               /wiki UI · /api/wiki/search · agent retrieval
```

**Key principle**: an `Ingestor` is a single TypeScript interface. Each
source kind implements `fullSync()` and `incrementalSync(cursor)` and
delivers normalized `WikiDocument` shapes. The downstream chunker / FTS /
embedding / page builder is source-agnostic — adding a new connector is
implementing one interface, not changing the wiki engine.

## Ingestor interface

```ts
export interface Ingestor {
  kind: WikiSourceKind;
  fullSync(source: WikiSourceRow, onDoc: (doc: WikiDocumentInput) => Promise<void>): Promise<{ ingested: number }>;
  incrementalSync?(source: WikiSourceRow, cursor: unknown, onDoc: (doc: WikiDocumentInput) => Promise<void>): Promise<{ ingested: number; nextCursor: unknown }>;
}

type WikiDocumentInput = {
  externalId: string;
  title: string;
  bodyMd: string;
  contentHash: string;
  metadata: Record<string, unknown>;
};
```

## Phase plan

### Phase A — Foundation + Tables source (this session)
- [ ] Schema: all six tables + trigger functions for tsv columns
- [ ] Ingestor interface + chunker + embedder helper
- [ ] TablesIngestor: wraps existing `docs` table → `wiki_documents` → `wiki_chunks`
- [ ] ConceptBuilder (Loop 5): cluster tables by FK + skill triggers, LLM writes concept pages with `[[slug]]` refs
- [ ] IndexBuilder: one rendered index page linking to concepts + leaves
- [ ] Link resolver: turn `[[slug]]` into `wiki_links` rows + render as `<a>` in UI
- [ ] Hybrid search API: FTS rank + pgvector cosine (when embeddings exist)
- [ ] `/wiki` UI: tree nav (Sources → Concepts → Leaves), markdown viewer with resolved links, backlinks panel, provenance panel, search box
- [ ] Scheduler hook: when Loop 1 detects schema delta or a new view/skill lands, mark related pages stale → next tick regens

### Phase B — Document upload connector (next session)
- File upload endpoint (multipart), stored in `loom-wiki-files/`
- Parsers: `pdf-parse` (PDF), `mammoth` (DOCX), `marked` (MD/TXT)
- Chunker: target ~700 tokens, 60-token overlap, respect headings
- Optional OCR pass for image-only PDFs (defer behind env flag)
- UI: `/wiki/upload` route with drag-drop

### Phase C — GitLab connector (next session)
- Auth: project access token, encrypted at rest
- Walk strategy: configurable globs (`README.md`, `docs/**/*.md`, `*.md`)
- Incremental: webhook receiver → enqueue ingest job; fallback polling on `last_commit_sha`
- Path-aware chunking + code-block awareness

### Phase D — SharePoint via MCP (next session)
- MCP client: attach a SharePoint MCP server (user provides the connector binary or hosted URL)
- OAuth via Microsoft Graph (delegated app, on-behalf-of flow)
- Sync drives/sites/libraries; delta token for incremental
- Note: we attach an EXTERNAL MCP rather than rebuild SharePoint auth from scratch — same pattern Loom uses for itself

### Phase E — Enterprise polish (next session)
- Secrets at rest: KMS-style envelope (pgcrypto for v0.4, real KMS in v1.0)
- Per-source ACL: `wiki_acl(source_id, principal, action)`; UI permission gates
- Background queue migration: in-process → graphile-worker
- Observability: structured logs with source_id correlation, per-source metrics
- Audit log: every read of a wiki page (`actor='user', action='wiki:view'`)
- Soft-delete on documents + page archival

## Open questions

- (resolved) Concept clustering algorithm: starting with FK graph + skill
  trigger overlap. If too coarse, layer in query-co-occurrence weights from
  `audit_log` in v0.4.1.
- (resolved) Embedding provider: OpenAI `text-embedding-3-small` if key
  present; otherwise FTS-only. Local embedding (e.g. `gte-small` via
  Voyage/Ollama) deferred to v1.0 per CLAUDE.md anti-scope.
- (open) Editing: how do human edits survive regen? — Same provenance pattern
  as table docs: paragraph-level tags, regen preserves `human` blocks. UI
  shows "edited X mins ago by Y" badge.

## Complexity

- Phase A: L (schema + ingestor + 2 generators + UI + search)
- Phase B: M (parsers are mature, chunking is tricky)
- Phase C: M (incremental sync correctness is the work)
- Phase D: M (MCP wiring is mechanical, OAuth scoping is the work)
- Phase E: L (cross-cutting; touches everything)

## Demo arc (when Phase A ships)

1. Open `/wiki` → see the Loom-generated index: "Production lifecycle",
   "Quality methodology", "Equipment & maintenance", plus links to all
   tables, views, and skills.
2. Click "Production lifecycle" → a concept page synthesized from
   `production_runs`, `operators`, `equipment` docs, with `[[deviations]]`
   and `[[deviation-rate]]` (skill) inline.
3. Type into the search box: `"why is LINE-B's deviation rate so high"` →
   hybrid results across leaf docs, concept pages, and (in Phase B) the
   methodology PDF you uploaded.
4. Hover any `[[skill:deviation-rate]]` link → tooltip with the SQL template.
5. Click "Edit" on a page → textarea, save → `human` provenance preserved on
   next regen.
